## Overview

This project tackles thread-level phishing detection — a harder problem than single-email detection because attacks span multiple emails designed to look legitimate. Four attack strategies are modeled:

| Attack Type | Strategy |
|---|---|
| **Thread Hijacking** | Attacker impersonates a known participant mid-thread and requests a wire transfer |
| **Relationship Exploitation** | Builds trust over several early emails, then exploits it |
| **Context Manipulation** | Fabricates references to prior meetings or conversations that never happened |
| **Urgency Escalation** | Starts with mild urgency and escalates pressure over the thread |

Attack threads were synthetically generated by three LLMs: **Claude** (205 threads), **ChatGPT** (95 threads), and **Grok** (100 threads). Legitimate threads come from the Enron email corpus (FERC investigation, 2001–2002).

## Dataset Setup

The Enron fraud dataset CSV is not included in this repository due to its size. Download it and place it at:

```
data/features/enron_data_fraud_labeled.csv
```

The CSV must have at least these columns: `Message-ID`, `From`, `To`, `Subject`, `Date`, `Body`, `Label` (0 = legitimate, 1 = fraud/spam).

The attack thread JSON files (`data/raw/Dataset Attack/`) are already included in the repository.

## Running the Pipeline

All scripts use **relative paths from their own directory**, so run each script from inside its folder.

### Step 1 — Explore the Dataset

Get a feel for the raw CSV structure, label distribution, and sample emails.

```bash
cd scripts/pipeline
python step1.py
```

**Output:** printed statistics only (no files written).

---

### Step 2 — Prepare Enron Emails

Filters for legitimate internal `@enron.com` emails, cleans addresses, and normalises subjects (strips `Re:`, `Fw:`, etc.).

```bash
cd scripts/pipeline
python step2.py
```

**Output:** `data/raw/enron_prepared/enron_emails_for_threads.json`

---

### Step 3 — Extract Email Threads

Groups emails into threads by **subject AND participant set**. Requires at least 3 emails and 2 unique senders per thread. Calculates a quality score (0–100) for each thread.

```bash
cd scripts/pipeline
python step3.py
```

**Output:**
- `data/raw/extracted_threads_v2/all_threads.json`
- `data/raw/extracted_threads_v2/high_quality_threads.json` (quality ≥ 50)
- `data/raw/extracted_threads_v2/individual_threads/thread_XXXX.json` (one file per thread)

---

### Step 4 — Deduplicate and Filter

Removes duplicate emails (MD5 fingerprint on sender + recipient + subject + date + first 300 chars of body), filters implausible dates, and verifies genuine back-and-forth conversation. Final filter keeps threads with quality ≥ 60, capped at 30 emails each.

```bash
cd scripts/pipeline
python step4.py
```

**Output:**
- `data/raw/cleaned_threads_v2/deduplicated_threads.json`
- `data/raw/cleaned_threads_v2/top_200_threads.json`
- `data/raw/cleaned_threads_v2/top_1000_for_dataset.json` ← used in Step 5

---

### Step 5 — Combine Dataset and Create Splits

Merges the 1,000 legitimate threads with all attack batches (400 threads across 4 attack types). Creates a stratified 70/15/15 train/val/test split.

```bash
cd scripts/pipeline
python combine.py
```

**Output:**
- `data/splits/combined_dataset.json` — all 1,400 threads
- `data/splits/train_dataset.json` — 980 threads
- `data/splits/val_dataset.json` — 210 threads
- `data/splits/test_dataset.json` — 210 threads

---

### Step 6 — Extract Features

Computes 12 thread-level features for every thread in each split. See [Features Explained](#features-explained) for what each one captures.

```bash
cd scripts/training
python extract_features.py
```

**Output:**
- `data/features/train_features.json`
- `data/features/val_features.json`
- `data/features/test_features.json`

---

### Step 7 — Train Baseline Random Forest

Trains a Random Forest (100 trees, max depth 10, balanced class weights) on the 12 features and evaluates on the test set. Saves the trained model.

```bash
cd scripts/training
python train_random_forest.py
```

**Output:**
- `models/random_forest_model.pkl`

## Features Explained

The 12 features are split across four categories:

**Temporal (3)**

| Feature | Description |
|---|---|
| `response_time_variance` | Variance (in hours) between consecutive email timestamps |
| `thread_length_days` | Total duration from first to last email in the thread |
| `timestamp_anomalies` | Fraction of emails sent between 23:00 and 05:00 |

**Linguistic (3)**

| Feature | Description |
|---|---|
| `tone_shift` | Absolute change in formality score from first to last email |
| `formality_variance` | Variance in formal vs. informal word usage across the thread |
| `urgency_escalation` | Change in urgency keyword density from first to last email |

**Coherence (3)**

| Feature | Description |
|---|---|
| `reference_accuracy` | Density of conversational references ("as mentioned", "per our call") |
| `entity_consistency` | Inverse of the number of unique email addresses in the thread |
| `topic_coherence` | Inverse of the number of unique subject base strings |

**Context-Aware (3)**

| Feature | Description |
|---|---|
| `fabricated_history_score` | Density of references to prior communications that cannot be verified in the thread |
| `relationship_velocity` | Number of formality shifts between consecutive emails |
| `cross_reference_count` | Density of references to prior thread content (forwards, attachments, CC mentions) |

## Results

**Best ML model — Gradient Boosting:**

| Metric | Score |
|---|---|
| Accuracy | 98.10% |
| F1 | 0.9667 |
| AUC-ROC | 0.9971 |

**Baseline Random Forest:**

| Metric | Score |
|---|---|
| Accuracy | 97.14% |
| F1 | 0.9202 |
| AUC-ROC | 0.9966 |

**Feature efficiency (RFE finding):** The top 5 features (`entity_consistency`, `thread_length_days`, `urgency_escalation`, `response_time_variance`, `formality_variance`) achieve 99.97% of the performance of all 12 features combined.
